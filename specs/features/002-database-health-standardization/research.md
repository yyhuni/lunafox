# 研究记录：数据库健康标准化接入

## 决策 1：健康状态判定归属

- 备选：
  - A. 前端继续根据阈值本地计算状态
  - B. 后端统一计算状态，前端仅渲染
- 结论：**选择 B**
- 原因：
  - 避免多端阈值漂移与语义冲突
  - 便于审计和统一运维口径
  - 降低前端复杂度

## 决策 2：`offline` 语义边界

- 备选：
  - A. 指标严重超阈值即 `offline`
  - B. 仅探活/连接失败判定为 `offline`
- 结论：**选择 B**
- 原因：
  - “不可用”与“性能下降”在运维场景必须分离
  - 指标超阈值应归类为 `degraded`

## 决策 3：核心指标与可选指标分层

- 备选：
  - A. 全部指标等权
  - B. 核心指标决定状态，可选指标仅补充观察
- 结论：**选择 B**
- 核心指标：
  - 探活延迟
  - 连接利用率（used/max）
  - 锁等待会话数
  - 死锁速率（1h）
  - 长事务数量（> 60 秒）
  - 最老待处理任务年龄
- 可选指标：
  - QPS
  - WAL 24h 增量
  - 缓存命中率
- 原因：
  - 确保状态判定稳定，减少权限不足或采集失败导致的误报

## 决策 4：时间字段规范

- 备选：
  - A. 后端直接返回“3m ago”等展示字符串
  - B. 后端返回 ISO 8601，前端本地化展示
- 结论：**选择 B**
- 原因：
  - 支持多语言和时区一致性
  - 支持排序与自动化分析

## 决策 5：后端落点策略

- 备选：
  - A. 新建完整 system 模块（router/handler/application/repository/domain）
  - B. 在现有 asset 模块中增量扩展 system/database-health 能力
- 结论：**当前阶段选择 B**
- 原因：
  - 现有健康检查已位于 `asset` 模块，增量成本更低
  - 可快速交付并验证行为
  - 若后续 `/api/system/*` 能力增长，再演进为独立模块

## 决策 6：部分失败策略

- 备选：
  - A. 任意指标采集失败即接口整体失败
  - B. 核心指标优先，允许可选指标缺失并显式报告
- 结论：**选择 B**
- 原因：
  - 提升故障期间可见性
  - 防止单点统计失败导致页面完全不可用

## 决策 7：前端刷新与故障反馈策略

- 备选：
  - A. 请求失败时清空页面
  - B. 有历史快照则保留并标记陈旧；无快照则展示错误态
- 结论：**选择 B**
- 原因：
  - 提高故障排障可读性
  - 避免“数据消失”带来的误导
